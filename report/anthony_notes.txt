--- Inference Bug Fixes (session 3) ---

Root cause of all-identical submissions:
- max_new_tokens was 192 but average training SVG is ~2524 chars (~600-1200 tokens).
  Every output was truncated before </svg>, so extract_svg returned nothing,
  is_valid_svg failed, and 100% of outputs fell back to the white+black-circle placeholder.
- Result: submission1.csv and submission_qwen3b_svg_lora_1774580690.csv were byte-identical
  (md5: 4434a81027e8224054aacadd07d37e50) — both all-placeholder.

Fix 1 — max_new_tokens: 192 -> 800
  Covers ~75th percentile of training SVG token lengths.

Fix 2 — Double-wrapped LoRA:
  Model load cell was calling get_peft_model even when loading a saved adapter,
  stacking a new untrained LoRA on top of the trained one. This caused the RoPE
  shape mismatch crash:
    RuntimeError: output with shape [1,16,1,128] doesn't match broadcast shape [1,16,43,128]
  Fix: only call get_peft_model when loading_saved_model is False (training from scratch).

Fix 3 — use_cache:
  use_cache=True causes the RoPE shape mismatch with Unsloth PEFT models. Reverted to False.
  (Note: use_cache=False is slower but required for PEFT-wrapped Unsloth inference.)

Fix 4 — batch_size: 16 -> 4
  With 800 new tokens per sample, batch KV-cache at size 16 would OOM on 9.6GB RTX 3080.
  4 x 800 ≈ 16 x 192 in terms of token budget per batch.

Fix 5 — Crash-safe incremental inference:
  Submission loop writes to {SUBMISSION_PATH}.part after each batch.
  On restart, resumes from already-done IDs in the .part file.
  os.replace() atomic rename at completion to final CSV.

Fix 6 — Dynamic submission filename:
  SUBMISSION_PATH = f"./submission_{model_dir_name}_{timestamp}.csv"
  Prevents overwriting previous submissions.

Saved Qwen 3B adapter: artifacts/qwen3b_svg_lora/
  Loads correctly with FastLanguageModel.from_pretrained (skip get_peft_model).
  For inference-only run: skip training cells, run cells 9 (model load) then 11-12.

--- SVG Regex False Match Bug (session 4) ---

Bug: SVG extraction was still returning the circle fallback even after max_new_tokens fix.
Root cause: the system prompt contains the literal text "<svg> element." — the regex
  r"<svg[\s\S]*?</svg>" matched that fragment as the opening <svg> tag, then grabbed
  everything up to the real SVG's closing </svg>, producing garbled XML that failed
  ET.fromstring with:
    ET.ParseError: no element found: line 5, column 246
  This caused is_valid_svg to return False and the fallback to trigger on every row.

Fix: decode only the new (generated) tokens, not the full input+output sequence.
  Before: tokenizer.decode(output_ids[0], skip_special_tokens=True)
  After:  tokenizer.decode(output_ids[0, input_len:], skip_special_tokens=True)
  where input_len = inputs["input_ids"].shape[1] (captured before model.generate).
  Same fix applied to generate_svg_batch: output_ids[:, input_len:]
  The system prompt text never appears in the decoded string, so the regex matches
  only the real SVG output.

Also noted: use_cache=False + batch_size OOM tradeoff.
  use_cache=True causes RoPE shape mismatch with this Unsloth PEFT model — must stay False.
  use_cache=False causes quadratic VRAM growth per step (no KV cache reuse).
  batch_size=4 OOM'd; batch_size=2 is the current setting (halves peak VRAM vs 4).
